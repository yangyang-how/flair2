Force 1

Work is slow.

260 LLM calls × 2–5s each. Can't hold an HTTP request open for minutes.

Concurrent tasks coordinate via atomic counters + SETNX. No locks.

The counter pattern (distributed barrier)

INCR is atomic. 100 concurrent workers → unique sequence numbers.
At 95%, SETNX guards the transition — exactly once, even under races.

Stream, not pub/sub

Streams persist, support cursors, replay. Reconnect mid-run → Last-Event-ID resumes from exactly that cursor. Multi-tab safe, zero extra code.

Near-linear speedup

42 IO-bound tasks complete in one latency unit instead of 42. Sequential S4 would add 4+ seconds per run.

Amdahl ceiling at cap = N

Beyond 29 concurrent slots, more workers buy nothing. The right optimization is Kimi's cap, not more ECS tasks.

★ Experiment 2

Free in the common case, huge in the worst

True bottleneck: Redis connection pool

Worker CPU stayed at 7%. Queue depth stayed at 0–1. /api/runs p99 spiked to 20s at K=500. The API's Redis pool is the choke point, not Worker capacity.

CPU is the wrong Worker autoscaling metric

Workers wait on LLM responses — CPU never exceeds 8%. Queue depth (LLEN celery) is the correct signal.

07 · Lessons

What broke.
What it taught.

Ten production incidents, condensed. Process over talent.

Ten incidents

01Gemini 500s + rate limits→ switch to Kimi (registry paid off)
02S1 always 500s in prod→ .dockerignore excluded data/ (4 PRs)
03Kimi all requests fail with weird error→ migrate OpenAI SDK → Anthropic Messages
04Workers deployed, pipeline silently hangs→ Celery never imported tasks module
05Frontend faked progress when stuck→ stall detector + API probe diagnosis
06"ELB already exists" on every deploy→ Terraform state drift → idempotent import
07S4 fails at persona ~40 with 429→ semaphore + retry budget per error class
08One bad video kills whole S1 run→ skip-and-continue + 95% threshold
09Pipeline hangs at 99/100→ ceil(N × 0.95) transition with SETNX
10"Start Pipeline" bounces to home→ S3 routing: path → query param

Process discipline that held

Hard branch protection

GitHub ruleset physically blocked direct pushes. 112 merged PRs, all cross-reviewed.

Interface contract up front

Every Redis key, every SSE event specified before writing the code. Let two devs ship in parallel for weeks.

Test failure paths explicitly

M5-2 validated checkpoint recovery before any real crash. First crash saved 50% of calls.

Single-writer discipline

Orchestrator is the only code path that mutates run state. No races, no locks.

08 · Business case

We don't guess.

Three pillars, each grounded in real data. Base-rate improvement, not a guarantee.

Pillar 1

Learn from winners.

S1 extracts structural patterns from 100 real viral TikToks. Every generated script is shaped by what already went viral.

Pillar 2

Test with the right crowd.

300 personas matching Canadian demographics + adoption. At run-time, pick the 100 most relevant to the creator's niche.

Pillar 3

Close the loop.

Posted videos report real outcomes. Next run's prompts calibrate toward what actually worked. Reality is the final judge.

01 · Problem

Flair2 — AI Campaign Studio

Generate social media marketing campaigns from viral video analysis. One user click triggers ~260 LLM calls and live streams the work. This shape forces a distributed system.

What the user sees

Enter a creator profile. Watch a 6-stage AI pipeline analyze 100 viral TikTok videos, generate candidate scripts, simulate 42 personas voting, and personalize the winners to the creator's voice. All in real time.

Three design forces

  • Work is slow. 260 calls × 2–5s each = minutes, not milliseconds. Can't hold an HTTP request open that long.
  • User needs live progress. A 15-second spinner is unacceptable. Must push events as each stage completes.
  • Work fans out per request. S1 analyzes N videos concurrently. S4 has M personas voting concurrently. Need MapReduce coordination inside a single request.
Every architectural decision that follows is a response to one of these three forces. No force → no need for the pattern.

02 · Architecture

Three Layers of Distribution

Each layer solves a different problem. Understanding which layer is hurting is half of operating the system.

Layer 1 — Edge ↔ API

Browser talks to ALB over HTTPS. ALB distributes to healthy API tasks. Solves availability + load balancing.

Layer 2 — API ↔ Workers

API enqueues to Redis, returns in milliseconds. Workers pull tasks and run them elsewhere. Solves long-running work.

Layer 3 — Worker ↔ Worker

Multiple workers process tasks for one run concurrently. Coordinate via atomic Redis counters + SETNX transition guards. Solves MapReduce inside one request.

03 · Pipeline

Six Stages, Two MapReduce Cycles

S1/S4 fan out (map). S2/S5 aggregate (reduce). S3 generates scripts sequentially. S6 personalizes top results.

REDUCE · 0 LLM calls · Python only

Personalize
Rewrite top-N scripts in creator's voice. Generate video prompts.

Creator profile shapes S3 + S6

The creator's niche, content themes, and recent topics are injected into S3's prompt, so scripts come out on-topic. S6 then personalizes the winners with the creator's tone, vocabulary, and catchphrases.

Before this fix: Kitchen Confident got productivity scripts in cooking vocabulary. After: actual on-topic cooking content.

Resilience baked in

  • Skip-and-continue: one un-analyzable video doesn't kill the run — it's marked skipped and S2 aggregates the rest.
  • 95% completion threshold: stragglers don't block transitions. S1→S2 fires when 95 of 100 complete. SETNX guards exactly-once dispatch.
  • S4 checkpointing: every vote writes a checkpoint. Crash mid-S4 resumes from there — saves 30-73% of LLM calls.
  • Silent failure is not allowed: if workers stall for 30s, frontend probes the API and tells the user specifically why.
Total shape: 260 LLM calls → 15 seconds on 20-way concurrency. Limited by Kimi's concurrency ceiling, not our CPU or network. The semaphore + retry-budget-per-error-class (next section) keeps us inside that ceiling.
04 · Coordination

Redis as the Nervous System

Every coordination primitive goes through Redis. Single-writer orchestrator + atomic counters + SETNX guards = MapReduce without locks.

The counter pattern (distributed barrier)

# When an S1 task completes:
done = await redis.incr(f"run:{id}:s1:done")

# Check if threshold crossed:
needed = math.ceil(num_videos * 0.95)
if done >= needed:
    # SETNX ensures exactly-once transition
    if await redis.setnx(f"run:{id}:s2:triggered", "1"):
        s2_aggregate_task.delay(run_id)

INCR is atomic — 100 concurrent workers never collide. SETNX ensures only one of them fires the transition.

The semaphore (concurrency limit)

# Before every Kimi call:
async with _acquire_provider_slot(redis, "kimi"):

# max_slots=29 — Kimi's ceiling is 30, we leave one slot

Any number of Celery workers can pull tasks. The semaphore caps in-flight LLM calls so we never trigger Kimi's concurrency throttle.

Single-writer orchestrator

Every state write and every SSE event goes through one code path — the Orchestrator. Workers report completions via callbacks (on_s1_complete, on_s4_complete); only the orchestrator mutates run:{id}:* keys or appends to the SSE stream. No locks, no races.

Same principle as: Raft leaders, event sourcing, database write-ahead logs, Node.js event loops. Mutation through one writer > mutation through many writers with coordination.

SSE via Redis Streams (not pub/sub)

Streams persist, support cursors, and handle replay. A browser that reconnects mid-run sends Last-Event-ID; the server resumes from exactly that cursor. Late-joining tabs replay from the beginning. Multi-tab safe with no extra work.

# SSE manager — blocks 5s for new events, then re-checks for disconnect
entries = await redis.xread({f"sse:{id}": last_id}, block=5000)
for entry_id, fields in entries:
    yield ServerSentEvent(id=entry_id, event=fields["event"], data=fields["data"])
    last_id = entry_id
05 · Kimi Integration

Two SDK Migrations, Zero Stage Changes

The provider abstraction paid for itself twice. When the LLM backend changed — both the provider AND the SDK — no pipeline code moved.

Migration 1 — Gemini → Kimi

Gemini had intermittent 500s and tight rate limits. Kimi Code offered better availability on a subscription plan. PR #95: one-line change in the registry, Terraform secret swap. Zero stage code touched.

Migration 2 — OpenAI SDK → Anthropic SDK

Kimi's OpenAI-compatible route went dead — started returning a misleading "temperature" error. The real surface moved to an Anthropic Messages API shape at /coding/v1/messages. Rewrote just the KimiProvider to use AsyncAnthropic. Zero stage code touched again.

Retry budget per error class

Most 429s aren't quota exhaustion — they're transient concurrency throttling that clears in 10–30 seconds. A fast retry budget (1s/2s/4s total 7s) is too impatient. A slow budget is wrong for parse errors. So we split them:

Fast budget — schema/parse errors

3 attempts · 1s / 2s / 4s

Transient noise. Retry aggressively. Total ~7s.

Slow budget — 429 rate limits

4 attempts · 8s / 20s / 45s / 90s + ±30% jitter

Real throttling. Be patient. Total ~165s at provider + ~5 Celery retries = ~10 min of patience.

Jitter prevents thundering herd. When 42 S4 voters all hit the rate limit within milliseconds and all retry on the same backoff schedule, they all get throttled again. ±30% jitter desynchronizes them.

The User-Agent whitelist

Kimi's coding endpoint is gated by User-Agent — only approved clients (Kimi CLI, Claude Code, Roo Code) can use it. We spoof User-Agent: claude-code/0.1.0 via default_headers. This is fragile — Kimi could tighten validation at any time — but it's the documented way the ecosystem uses it.

Cross-experiment findings

  1. Redis connection pool is the system-wide bottleneck at scale. Both M5-4 (API p99 = 18s at K=500) and M6-2 (SETNX fails at K=5000) point to the same root cause — connection pool exhaustion. Fix is pool sizing, not instance scaling.
  2. CPU is the wrong auto-scaling metric for Workers. Worker CPU peaked at 7.14% under the heaviest load. Workers are IO-bound on LLM calls. The correct signal is Celery queue depth.
  3. fakeredis hides real failure modes. M5-3 passed at K=100,000 in fakeredis. M6-2 failed at K=5,000 on ElastiCache. Connection pools are invisible in the simulator.
  4. Auto-scaling works but needs the right metric. ECS correctly detected CPU overload and added tasks. But CPU wasn't the bottleneck.
07 · Lessons

Things That Broke and What They Taught

Every incident left a fingerprint. Here are the ones worth carrying forward.

Process lessons

  • Hard branch protection works. GitHub ruleset physically blocked direct pushes and required a reviewer. 112 merged PRs, all reviewed. Neither of us could ship without the other reading the diff.
  • Interface contract (issue #71) up front. Every Redis key, every SSE event, every API route specified before anyone wrote the corresponding code. Let Sam and Jess work in parallel for weeks and integrate cleanly.
  • Conventional commits. feat: / fix: / chore: / docs:. Makes git history greppable. "What did we change about Celery?" is a one-line search.
  • Test failure paths explicitly. M5-2 validated checkpoint recovery before any real crash. The first production crash saved 50% of LLM calls on the first try.

Technical lessons

  • "X-compatible API" claims expire. Kimi's OpenAI-compatible shim went dead. The abstraction let us migrate to Anthropic's SDK in ~30 lines.
  • Connection pools are invisible bottlenecks. Tail latency spikes with nothing obviously saturated = bounded resource exhaustion. Instrument pools explicitly.
  • Retry budget per error class. Transient noise needs fast retries. Upstream throttling needs patience. One budget can't satisfy both.
  • Single-writer discipline beats locks. The orchestrator is the only code path that mutates run state. No races, no locks, no bugs.
  • Don't fake progress. If workers stalled, show the user specifically why — don't let the UI simulate activity that isn't happening.
  • Skip-and-continue > fail-the-run. At N=100 the probability of at least one bad input approaches 1. Design for partial failure from day one.

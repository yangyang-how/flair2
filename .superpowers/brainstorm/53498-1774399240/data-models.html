Section 2: Data Models — Pydantic Contracts Between Stages

Every stage has a typed input and output. These models ARE the spec — if it's not in the model, it doesn't exist.

DDIA Principle: Encoding & Evolution

Kleppmann: "Schema-on-write enforces structure at write time — early error detection." Pydantic models are schema-on-write. If S1 produces output missing a field, it fails immediately — not silently downstream in S5.

Error Hierarchy

# models/errors.py

class PipelineError(Exception):
    """Base error — all pipeline errors inherit."""
    def __init__(
        self,
        message: str,
        run_id: str | None = None,
        stage: str | None = None,
        attempt: int | None = None,
    ): ...

class ProviderError(PipelineError):
    """LLM/Video API failure — retryable."""
    provider: str
    status_code: int | None

class RateLimitError(ProviderError):
    """Rate limit hit — backoff and retry."""
    retry_after: float | None

class InvalidResponseError(ProviderError):
    """LLM returned unparseable output."""
    raw_response: str

class StageError(PipelineError):
    """Pipeline logic failure — halt stage."""
    pass

class InfraError(PipelineError):
    """Redis/S3/DynamoDB failure — alert."""
    service: str  # "redis" | "s3" | "dynamodb"

What This Means For Development

  • Stage functions are typed: def s1_analyze(input: VideoInput, provider: ReasoningProvider) -> S1Pattern
  • Tests are assertions on models: assert result.hook_type in ["question", "shock", "story", "direct"]
  • Serialization is free: result.model_dump_json() for Redis/S3/DynamoDB
  • Schema evolution is safe: Add optional fields with defaults — backward compatible (DDIA Ch. 4)
  • Invalid data fails immediately: Pydantic validation catches bad LLM output at the boundary, not 3 stages later

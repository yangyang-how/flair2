How should the backend modules be layered?

Dependencies flow downward only. Pick the layering that makes sense for this project.

A

4-Layer: API → Pipeline → Providers → Infra

Classic layered architecture. API routes call pipeline orchestrator. Pipeline stages call LLM providers. Providers and stages use infra clients (Redis, S3, DynamoDB). Clean separation but deeper call stack.

┌─────────────────────────────────┐
│  API Layer                      │  FastAPI routes, SSE, sessions
│  api/routes/*.py                │
├─────────────────────────────────┤
│  Pipeline Layer                 │  Orchestrator, stage functions,
│  pipeline/stages/*.py           │  prompts, Celery tasks
│  pipeline/orchestrator.py       │
├─────────────────────────────────┤
│  Provider Layer                 │  LLM + Video provider interfaces
│  providers/*.py                 │  Retry, rate limiting wrappers
├─────────────────────────────────┤
│  Infra Layer                    │  Redis, S3, DynamoDB, config
│  infra/*.py                     │  All AWS/external I/O lives here
└─────────────────────────────────┘
B

3-Layer: API → Core → Infra

Providers are part of the core pipeline layer (since stages call providers directly). Simpler, fewer modules, but the "core" layer is bigger and mixes concerns.

┌─────────────────────────────────┐
│  API Layer                      │  FastAPI routes, SSE, sessions
│  api/routes/*.py                │
├─────────────────────────────────┤
│  Core Layer                     │  Orchestrator, stages, providers,
│  pipeline/*.py                  │  prompts, Celery tasks — everything
│  providers/*.py                 │  that isn't API or infra
├─────────────────────────────────┤
│  Infra Layer                    │  Redis, S3, DynamoDB, config
│  infra/*.py                     │
└─────────────────────────────────┘
C

Hexagonal: Core (pure) + Ports + Adapters

Stage functions in the center are pure — no imports from providers or infra. They receive a provider as a parameter (dependency injection). Adapters (Redis, Gemini, S3) plug in from outside. Maximum testability, more abstract.

         ┌──── Adapters ─────┐
         │ GeminiProvider     │
         │ RedisStateStore    │
         │ S3FileStore        │
         │ DynamoMetaStore    │
         └────────┬───────────┘
                  │ implements
         ┌────────▼───────────┐
         │    Ports (Protocol) │
         │ ReasoningProvider   │
         │ VideoProvider       │
         │ StateStore          │
         │ FileStore           │
         └────────┬───────────┘
                  │ injected into
         ┌────────▼───────────┐
         │   Core (pure)      │
         │ s1_analyze()       │
         │ s2_aggregate()     │
         │ s3_generate()      │
         │ orchestrator       │
         └────────────────────┘

DDIA Principle: Maintainability

Kleppmann: "The majority of the cost of software is in ongoing maintenance." The layering choice determines how easy it is to swap providers, add new stages, write tests, and onboard Jess. Option C gives the most isolation but adds abstraction cost. Option A is the industry standard. Option B is simplest but may get messy as the codebase grows.
